Master Ledger of Implemented FixesUse this list to track the specific bugs that were neutralized and how the system behaves now.The Zeek Timing Cliff (Sync Gap): Zeek previously used a hard 300-second global reset, meaning threats detected near the boundary were wiped before Pi-hole correlation.  Fix: Converted static integer counters to time-aware sliding deque structures.  Improvement: Threat events age out smoothly, guaranteeing perfect cross-layer correlation and eliminating timing race conditions.Lateral Movement False Positives: Zeek blindly flagged internal SSH/RDP connections, even to trusted administrative servers.  Fix: Passed the live safe_ips configuration directly into the Zeek Extractor to silently ignore connections involving whitelisted infrastructure.  Improvement: Drastically reduces alert fatigue and prevents safe servers from being penalized.Missing Destination Ports in Alerts: Loki dashboards could not query specific ports because they were nested or dropped during alert generation.  Fix: Extracted dest_port from Zeek events and injected it as a top-level key in the JSON payload written to alerts_stream.jsonl.  Improvement: Enables direct Grafana dest_port filtering without complex JSON unpacking.Device Identity Split (Memory Leak): Devices generating DNS requests before Zeek registered their DHCP MAC address created fragmented tracking profiles.  Fix: Implemented dynamic state migration. When a MAC address is discovered, the IP-based state is seamlessly merged into the new MAC-based state.  Improvement: Prevents duplicate AI baselines and stops orphaned state objects from leaking RAM.Risk Flapping (Alert Spam): Devices bouncing between a risk of 5.9 and 6.1 bypassed the 60-second alert deduplication lock.  Fix: Added a -1.0 cooldown margin. The risk score must drop below 5.0 before the deduplication timer arms itself again.  Improvement: Completely stops Telegram alert spam for borderline anomalous devices.Safe-Device Probation Reset: Modifying the safe list purged devices from active memory, forcing them into a 24-hour baseline probation penalty if re-added.  Fix: Removed the memory purge function. Safe devices now stay in active memory to maintain warm AI baselines, but their final risk scores are forcibly overridden to 0.0.  Improvement: Safe devices instantly resume mature monitoring if their whitelist status is revoked.Poisoned Baseline Log Exhaustion: A frozen baseline generated a severity warning 30 times a minute per device.  Fix: Throttled the logger output for active infections to a maximum of 1 log per hour per device.  Improvement: Restores syslog readability and prevents storage exhaustion.Dead-Letter Memory Exhaustion: Failed Telegram API delivery attempts queued indefinitely.  Fix: Capped delivery retries at 25 attempts using an exponential backoff curve.  Improvement: Prevents the background worker thread from crashing due to memory bloat during prolonged internet outages.2. Core Logical Decisions & AlternativesWhen modifying the script in the future, reference these structural trade-offs to ensure new features align with the original design philosophy.Baseline Freezing vs. Continuous Adaptive LearningDecision: When a device's risk hits $\ge$ 7.0, baseline updates are frozen (is_poisoned = True).  Alternative: Allow the model to continuously learn during an active incident.Advantage: Prevents attackers from using "slow-and-low" campaigns to gradually train the AI that malware behavior is normal.Disadvantage: A severe false positive will freeze the baseline until the rolling time window clears the anomaly.Sliding Time Windows (deque) vs. Global Reset TimersDecision: Zeek events are tracked using capped collections.deque objects and pruned dynamically by timestamp.  Alternative: Clear all Zeek dictionaries universally every 300 seconds.Advantage: Completely eliminates timing-cliff race conditions; every event influences the score for exactly 300 seconds.Disadvantage: Higher memory footprint. (Mitigated by enforcing strict maxlen parameters on all deques).  State Migration vs. Early Traffic IgnoranceDecision: The engine tracks via IP first, then migrates historical data to a MAC state when Zeek resolves the DHCP binding.  Alternative: Ignore all device traffic until the MAC address is definitively known.Advantage: Zero data loss. Early connection anomalies (e.g., immediate malware beaconing upon joining the network) are caught instantly.Disadvantage: Requires complex, thread-safe dictionary mutation (states.pop) inside the active execution loop.  Asynchronous Context Enrichment vs. Blocking LookupsDecision: Resolving domain IPs and Geo-locations is handed off to a non-blocking ThreadPoolExecutor.  Alternative: Halt the main loop to run socket.gethostbyname().Advantage: The main pipeline maintains its strict 2.0-second Pi-hole polling interval without lagging or desyncing.Disadvantage: The first log line for a newly seen domain might momentarily report its GeoIP/ASN as "unknown" until the background thread completes.3. Future Integrity & Cross-Checking ChecklistRun through this reusable checklist before deploying future features to ensure the architecture does not break:The Deque Cap Check: If adding a new metric to ZeekFeatureExtractor, use a deque instead of an int. Apply a strict maxlen= parameter to prevent Out-Of-Memory (OOM) crashes during a DDoS attack.The Sync Prune Check: If a new deque is added in Zeek, add it to the prune() function so it successfully ages out of the sliding window.The State Lock Check: If manipulating the states OrderedDict (adding, removing, or migrating a device), wrap the operation cleanly inside a with STATE_LOCK: block to prevent threading collisions.The Safe Override Check: If adding a new alert trigger or IPS mitigation command, ensure it respects the if is_safe: bypass block (Safe servers must never get accidentally isolated).The Non-Blocking Check: If introducing an external API call (like a new Threat Intel feed), handle it asynchronously or in a background thread. Anything blocking the main loop >1.5 seconds will break Pi-hole synchronization.The Flapping Check: If introducing a new alert threshold, build in a negative recovery margin (e.g., -1.0) so the system does not spam alerts when the metric bounces on the borderline.
